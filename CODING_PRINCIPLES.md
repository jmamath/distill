# Distill Coding Principles

> **Purpose:** To guide human engineers and AI agents in writing readable, maintainable, and predictable code for this project.

## 1. Core Engineering Philosophies

### KISS (Keep It Simple, Stupid)
Prioritize simplicity and readability over cleverness or premature optimization. Write code that explains itself through its structure.

### Comments Explain the "Why", Not the "How"
Code demonstrates *how* a task is performed; comments must explain *why* it was done that way, providing necessary business or architectural context.

### Modularity & DRY (Don't Repeat Yourself)
Break down systems into distinct, reusable, and single-purpose components. Every piece of knowledge or logic must have a single, unambiguous representation.

### Design by Contract
Every function, method, and class must have an explicit contract defined by a clear docstring stating expected inputs, guaranteed outputs, side effects, and raised exceptions.

### File-Level Docstrings
Every file must have a top-level docstring explicitly describing in plain words why the file exists. If the file contains a `main` function or is a test file, this docstring must include exact instructions on how to run it.

## 2. Complementary Principles for AI-Agent Collaboration

### Strict Type Hinting
Every function, method signature, and complex data structure must be fully type-hinted to establish a verifiable, machine-readable contract.

- **Use built-in generic types** — prefer `list[str]`, `dict[str, int]`, `tuple[str, ...]` over `typing.List`, `typing.Dict`, `typing.Tuple`. The `typing` equivalents were deprecated in Python 3.9 (PEP 585).
- **Never use `Any`** — `Any` defeats type checking and obscures contracts. If a dict has a known shape, define a `TypedDict`. If a value can be one of several types, use a `Union` (`X | Y`). Reserve `Any` only as a last resort when wrapping truly untyped third-party boundaries, and add a comment explaining why no better type exists.
- **Imports at the top** — all imports must be grouped at the top of the file, ordered: standard library → third-party → local. Never import inside functions or conditionally, except to break circular imports (which should be resolved structurally instead).

### Explicit is Better than Implicit (Fail Fast)
Do not silently swallow errors or pass unexpected `None` values. If a contract is breached, raise a descriptive exception immediately.

### Isolate Side-Effects (Purity)
Separate pure, deterministic data-transformation logic from functions that execute side-effects (e.g., file operations, network requests, database access).

### Descriptive, Intention-Revealing Names
Variables and functions should fully describe their intent and purpose (e.g., `fetch_user_profile_by_email` over `get_data`) to act as built-in semantic documentation.

### Human Oversight Legibility

Agents have broad knowledge but no stake in the outcome. Humans take the risk. Any artifact an agent writes for human review — tests, plans, documentation, runbooks — must answer **"why should I care?"** before asking a human to act on it.

In practice:
- **Test steps** must include a `> **Why:** ...` line explaining the user-visible or business consequence if the check fails (see `docs/specs/README.md`).
- **Plan tasks** must open with the business motivation, not just what changes. If the task is purely technical, state what user-facing feature or risk it protects.
- **Documentation** must lead with context for someone unfamiliar with the area before diving into technical detail.

The bar: a non-expert human reading only this artifact should understand what they are risking by skipping or ignoring it.

### Commit Directly to Main
Agents must write and commit changes directly to the `main` branch, not to worktree or feature branches. Committing to a side branch forces a merge step that can introduce conflicts and obscures the change history. If a worktree is active, apply edits to the files tracked by `main` and commit there.

### Never Commit Without Explicit Instruction
Agents must not create git commits unless the human explicitly asks for one. The human must review all changes before they are committed. Write files, run tests, and report results — then wait. Do not stage or commit anything autonomously.

### No Agent Self-Promotion
Agents must not add signatures, attributions, or co-author lines to commits, pull requests, or any project artifact. Commit messages, PR descriptions, code comments, and documentation must read as if written by the team — not by the tool that generated them. No `Co-Authored-By:`, `Generated by`, or similar lines.

### Logging Format
Every file that calls `logging.basicConfig` (i.e. entry points and `__main__` blocks) must use the standard format:
```python
logging.basicConfig(level=logging.INFO, format='%(asctime)s - %(name)s - %(levelname)s - %(message)s')
```
All other modules must only call `logger = logging.getLogger(__name__)` and never call `basicConfig` themselves — format is set once at the entry point and inherited everywhere.
